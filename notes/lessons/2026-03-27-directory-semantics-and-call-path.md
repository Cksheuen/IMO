# Lesson: 目录设计必须同时定义语义与调用链

- Date: 2026-03-27
- Trigger: 本次对 `hooks/` 与 `notes/` 的重新设计

## 现象

仓库里 `hooks/` 与 `notes/` 长期没有文件。

## 错误认识

只从“目录现在是空的”推导“缺少设计说明”，但没有继续追问：

- 这个目录应该由谁写入？
- 在什么时机写入？
- 是否有运行时配置把它接上？

同时，上一轮设计还犯了另一个错误：没有先把 `.gitignore` 注释当作一级语义锚点，而是直接给 `notes/` 做了过窄的二级定义。

## 根因

目录设计只回答了“是什么”，没有回答“如何被调用”。

这会导致两个后果：

1. 语义漂移：设计者容易用自己的抽象覆盖原始定义
2. 调用缺失：目录即使存在，也不会自动产生内容

## 正确做法

每次设计目录，都要同时回答四个问题：

1. 一级语义锚点是什么
2. 这个目录放什么，不放什么
3. 谁负责写入 / 读取
4. 触发条件和配置入口是什么

对这次问题，正确答案是：

- `hooks/` 的锚点来自 `.gitignore`：自定义 hooks（事件钩子脚本）
- `notes/` 的锚点来自 `.gitignore`：知识沉淀（经验教训、笔记）
- `hooks/` 的调用链来自 `settings.json` 或项目级 `.claude/settings.json`
- `notes/` 的调用链来自 `CLAUDE.md` / skills / 日常工作流中的写入触发条件

## 防复发规则

- 先读目录注释，再做二级抽象
- 设计目录时必须补“调用链”章节
- 如果一个目录长期为空，先查调用链是否缺失，再查语义是否缺失
- 对 loop / hook / 自动流程，必须额外补“真实载入要求”：挂载位置、触发事件、消费方、验证方法
- 只要缺少真实挂载或验证证据，就不能写成“已落地”“自动触发”“当前运行时协议”
