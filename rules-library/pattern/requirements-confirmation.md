# 需求确认规范

> 来源：`notes/lessons/implementation-vs-user-intent-mismatch.md` + Superpowers/brainstorming 启发 | 吸收时间：2026-03-31

## 核心问题

模型在未完全理解需求时就开始工作，导致：
1. 代码冗余（写了不相关的代码，未清理）
2. Token/时间浪费（用户需重复纠正）

## 核心原则

**确认优于假设，一次性问完**

| 原则 | 含义 |
|------|------|
| **Hard Gate** | Complex 级别禁止未经批准的实现 |
| **批量提问** | 所有问题一次性列出，不反复打断 |
| **分级处理** | 不同复杂度用不同确认强度 |
| **预设选项** | 多选格式，减少用户输入 |

## 触发条件

当任务满足以下任一条件时，执行确认流程：

- 需求存在多种合理理解
- 用户明确区分“参考 vs 复用”“调整 vs 新建”等边界
- 任务达到 Moderate 或 Complex 复杂度

## 复杂度分级

| 级别 | 标准 | 确认流程 |
|------|------|---------|
| **Trivial** | 单行修复、错别字、明确的小调整 | 直接执行，无需确认 |
| **Simple** | 目标明确，1-2 文件，无歧义 | 直接执行；如存在轻微歧义，可问 1 个确认问题 |
| **Moderate** | 多文件，或存在 1-2 个歧义点 | 需求复述 + ≤ 3 个问题 |
| **Complex** | 目标模糊，架构选择，≥ 3 个歧义点 | 完整确认流程（见下） |

## 执行流程

### Step 1: 复杂度评估

```
任务输入 → 评估复杂度
    │
    ├─ Trivial → 直接执行
    ├─ Simple → 仅在存在歧义时确认 → 执行
    ├─ Moderate → 确认流程（简化）→ 执行
    └─ Complex → 确认流程（完整）→ 执行
```

**复杂度评估检查点**：
- [ ] 涉及多少文件？
- [ ] 需求是否存在多种理解？
- [ ] 是否涉及架构决策？
- [ ] 是否需要修改现有代码？（参考 vs 复用）

### Step 2: 歧义检测（Moderate/Complex 必须）

**常见歧义类型**：

| 歧义类型 | 示例 | 确认问题 |
|---------|------|---------|
| **参考 vs 复用** | "参考 X 实现" | 是提取原理重新设计，还是复用代码？ |
| **调整 vs 新建** | "处理一下训练脚本" | 修改现有脚本，还是创建新脚本？ |
| **范围边界** | "优化性能" | 优化哪些指标？可接受的代价？ |
| **实现优先级** | "加上用户认证" | 先实现哪种认证方式？ |

### Step 3: 需求复述（Moderate/Complex 必须）

**模板**：

```markdown
## 我理解的需求

**目标**：[一句话描述]

**范围**：
- 包含：...
- 排除：...

**关键假设**：
1. [假设1]
2. [假设2]

**待确认**（请一次性回答）：
1. [Blocking 问题，必答]
2. [Blocking 问题，必答]
3. [Preference 问题，可选]
```

**一次性提问规则**：
- 所有问题在**一个消息**中列出
- 问题数量 **≤ 5 个**
- 超过 5 个问题 → 需求本身太模糊，建议先拆分
- Blocking 问题和 Preference 问题分开标注

### Step 4: 方案选项（Complex 必须）

当存在多种实现路径时，**主动**提出 2-3 个方案：

```markdown
## 实现方案

**推荐方案**：[方案名]
- 理由：...
- 代价：...

**备选方案**：
- 方案 B：...（优缺点）
- 方案 C：...（优缺点）
```

### Step 5: 确认许可（Complex 必须）

**硬门控**：Complex 级别必须等待用户明确批准。

```
用户批准？
    ├─ "可以"/"开始"/"执行" → 开始实现
    ├─ "方案 A" → 按选定方案实现
    ├─ 修改意见 → 更新理解 → 重新确认
    └─ 无响应 → 不执行，等待
```

**禁止行为**：
- ❌ 假设沉默 = 同意
- ❌ 理解偏差后"先做了再说"
- ❌ 只问一个问题就假设其他都清楚了

## 反模式

| 反模式 | 正确做法 |
|--------|----------|
| 问一个问题 → 等回答 → 又问一个 | 一次性列出所有问题 |
| "我理解了"就开始写代码 | 必须输出需求复述 + 等待确认 |
| 假设用户说"调整"就是"修改现有" | 区分"参考"与"复用"、"调整"与"新建" |
| "这个很简单，不需要确认" | Simple 级别可选确认，Moderate+ 必须确认 |
| 沉默就继续执行 | Complex 级别必须等待明确批准 |

## 检查清单

### Moderate 级别

- [ ] 是否输出了需求复述？
- [ ] 是否一次性列出所有问题？
- [ ] 问题是否 ≤ 3 个？
- [ ] 是否区分了 Blocking 和 Preference？

### Complex 级别

- [ ] 是否输出了需求复述 + 关键假设？
- [ ] 是否提出了 2-3 个方案选项？
- [ ] 是否等待用户明确批准？
- [ ] 批准前是否禁止了任何实现动作？

### UI 功能实现后（Feature Status 验证）

- [ ] 每个可交互元素是否有对应的 click/change handler？
- [ ] enabled/disabled 条件是否正确计算并覆盖所有状态？
- [ ] 内容可能超出容器时是否配置了 scroll？
- [ ] 是否从用户视角模拟了完整交互流程？
- [ ] 视觉表现与实际交互能力是否一致？
