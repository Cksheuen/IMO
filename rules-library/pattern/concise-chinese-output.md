# 中文简洁输出规范

> 来源：引入 [JuliusBrussee/caveman](https://github.com/JuliusBrussee/caveman) 本地魔改版时，将其哲学沉淀为规则基线 | 吸收时间：2026-04-17

## 核心问题

- `rules/core/user-facing-language.md` 规定"默认中文"，但未约束**输出长度与风格**
- Agent 容易在中文输出里保留英文 filler 模式（"好的我来帮你看看"、"这是一个不错的想法"）
- 复杂任务下即使 caveman 开关关闭，也应保留中文简洁基线
- 需要一条规则让"简洁"的语义独立于 caveman 协议存在

## 核心原则

**中文 + 无客套 + 技术精确**

三者共同构成面向用户的输出基线。关 caveman 只是取消"协议级强制注入"，基线规则仍然生效。

## 触发条件

面向用户的任何输出，包括：

- progress update / intermediary status
- 结论说明 / review findings
- 错误解释 / blocker 说明
- 解释命令、日志、报错、配置项含义
- 调研汇报、需求确认
- 操作建议、下一步动作

## 执行规范

### 1. 客套话黑名单（任何档位都删）

| 禁用 | 原因 |
|------|------|
| `好的我来帮你看看` | 纯 filler，无信息量 |
| `没问题` | 同上 |
| `希望这对你有帮助` | 结尾 filler |
| `这是一个不错的想法` | 无意义附和 |
| `让我先...再...然后...` | 冗长元描述，直接写动作即可 |
| `如有问题请告诉我` | 后续用户自然会问 |

### 2. Hedging 黑名单

| 禁用 | 正确做法 |
|------|----------|
| `也许` / `可能` / `我觉得` / `我认为` | 若不确定，直接列出未知量或用 `?` 标记 |
| `基本上` / `其实` / `某种程度上` | 直接去掉 |
| `应该可以` / `看起来像是` | 说明条件或前提，而不是模糊结论 |

### 3. 保留白名单

以下内容保留英文原样：

- 技术术语（polymorphism / idempotency / SIGTERM / bucketing）
- 代码、命令、路径、环境变量
- 报错原文、日志片段、堆栈
- API 字段名、类型名、协议名
- 引用的英文文档原句

英文对象**前后的解释文本**仍须中文。

### 4. 结构优先级

- **结论先给**：用户最关心"是/否"、"能/不能"、"路径在哪"
- **证据其次**：文件路径、行号、命令输出、引用来源
- **解释最后**：为什么这样做、有什么取舍

### 5. 长度控制

- 单次 progress update ≤ 25 字（除非在 allowlist skill 场景）
- 单次 final answer ≤ 100 字（除非任务要求详细展开）
- 表格 / 列表优先于段落
- 重复信息只说一次

## 豁免场景

以下场景暂停本规则，允许完整叙述：

- 用户显式要求详细解释
- `brainstorm` / `eat` / `orchestrate` / 设计讨论
- 破坏性操作确认、安全警告
- 多步不可逆序列（迁移、部署、回滚）
- 用户反复追问 → 说明简短版本没讲清楚，需展开

## 反模式

| 反模式 | 正确做法 |
|--------|----------|
| 英文 filler 翻成中文 filler | 整句删除，不保留 |
| `可能是因为 X` 包裹结论 | 结论直给，未知量单独列 |
| 每段都重述"我理解你想要..." | 直接给结果 |
| 用"首先/其次/最后"连接短句 | 用列表或直接并列 |
| 长段解释技术名词 | 用表格或类比，一行一条 |

## 检查清单

- [ ] 面向用户的中文输出里是否仍含 filler / hedging？
- [ ] 英文对象外是否包裹中文解释？
- [ ] 结论是否排在证据与解释之前？
- [ ] 是否有重复信息可以合并？
- [ ] 列表/表格是否可以替代段落？
