# Agent 概念流规范

> 来源：对“agent 交付结果之外，是否应顺带帮助用户学习关键开发概念”的调研收敛 | 吸收时间：2026-04-20

## 核心问题

Agent 很容易停留在：

- 汇报方案
- 输出代码
- 说明修复结论

但对刚接触相关主题的开发者来说，这类输出往往只解决“这次怎么做”，没有同步解释：

- 当前在用什么开发概念
- 为什么此处要这样判断
- 下次遇到类似问题时该如何迁移

结果是用户学会了“怎么跟 agent 协作”，却没有从 agent 的工作中形成稳定的知识体系。

## 核心原则

**教当前正在发生的概念，不做脱离上下文的百科解释。**

概念流必须同时满足：

1. **Context-bound**：概念必须绑定当前代码、当前设计、当前 bug
2. **Progressive disclosure**：默认只解释 1-2 个最相关概念
3. **Worked example first**：优先用当前例子讲，不先讲抽象定义
4. **Transfer-oriented**：解释完后要给出可迁移的判断句

## 触发条件

当满足以下任一情况时，建议启用概念流：

- 用户明确要求“讲讲原理 / 概念 / 为什么这样做 / 帮我理解”
- 用户表明自己刚接触相关内容，或明显处于 novice 状态
- 当前任务包含架构拆分、边界划分、调试思路、测试取舍、异步流程、状态建模等可迁移判断
- 当前回复除了交付结果外，补一小段概念说明能显著提升理解

## 禁用场景

以下场景默认不要启用：

- 纯 progress update / intermediary status
- trivial 修复、机械性替换、错别字调整
- 用户明确要求“只给结果，不要教学”
- 紧急排障或高压响应场景，此时应先解决问题

## 执行规范

### 1. 先交付，再教学

概念流是主任务的补充层，不可喧宾夺主。

输出顺序默认应为：

1. 结论 / 修改结果
2. 证据 / 文件 / 风险
3. 概念流（如适用）

禁止把概念解释写在前面，压住用户最关心的结果。

### 2. 每次只选 1-2 个概念

优先选择对当前任务最有解释力的概念，例如：

- 抽象边界
- coupling / cohesion
- invariant
- idempotency
- pure vs impure logic
- state machine
- sync vs async boundary
- cache invalidation
- retry / backoff
- interface contract

如果同时想解释 3 个以上概念，说明没有收敛，必须压缩。

### 3. 概念必须绑定当前例子

不要只说：

- “这是解耦”
- “这是状态管理”
- “这是幂等”

必须补上“在这里它具体体现在哪里”，例如：

- 为什么把校验逻辑从页面中抽出，属于 `UI / logic boundary`
- 为什么这个接口要幂等，和重试路径有什么关系
- 为什么这里用状态机比布尔组合更稳

### 4. 默认使用短结构

推荐骨架：

```markdown
## Concept
- 这次实际在用的是：...
- 它在这里重要，因为：...
- 这个改动里的体现：...
- 常见误解：...
- 下次遇到 ... 时，也可以用这个判断
```

默认只需 3-5 行，不必每次都写满全部字段。

### 5. 常见误解优先于长定义

若用户是新手，优先解释：

- 最容易误解什么
- 为什么该直觉在这里不成立
- 应该换成什么判断方式

这通常比给教科书定义更有帮助。

### 6. 用户显式要学习时，再升档

当用户明确进入学习模式时，可以扩展为：

- `Concept`
- `Why It Matters Here`
- `Worked Example`
- `Common Misconception`
- `Transfer`

必要时可追加一个简短 `self-check`，但默认不要求用户必须回答。

## 概念选择框架

优先顺序：

1. **当前决策背后的概念**
2. **当前错误暴露出的误解**
3. **下次仍可复用的判断**

避免选择：

- 与当前任务联系弱的术语
- 需要大段背景才说得清的理论
- 只能增加“懂很多”的表象、却不能指导行动的名词

## 决策框架

```text
这次回复是否值得加概念流？
    │
    ├─ 只是状态同步 / trivial 结果？
    │       └─ 是 → 不加
    │
    ├─ 用户明确要理解“为什么”？
    │       └─ 是 → 加
    │
    ├─ 当前任务含可迁移判断？
    │       ├─ 否 → 不加
    │       └─ 是 → 选 1-2 个概念
    │
    └─ 解释会不会压过结论本身？
            ├─ 会 → 压缩或取消
            └─ 不会 → 加短结构
```

## 反模式

| 反模式 | 正确做法 |
|--------|----------|
| 每次回复都附带概念讲解 | 仅在适合教学的场景启用 |
| 先讲理论，后给结果 | 先交付结果，再补概念 |
| 一次塞 4-5 个术语 | 收敛到 1-2 个最相关概念 |
| 只给术语定义，不结合当前代码 | 必须绑定当前例子 |
| 用概念流替代具体修改说明 | 概念流只能补充，不能替代结果 |
| 为了“显得专业”故意堆术语 | 优先讲误解、判断和迁移 |

## 检查清单

- [ ] 这次是否真的需要概念流，而不是普通结果汇报？
- [ ] 是否只选了 1-2 个最相关概念？
- [ ] 是否明确说明了“它为什么和当前任务有关”？
- [ ] 是否给了当前例子，而不是脱离上下文定义？
- [ ] 是否给出了一句可迁移判断？
- [ ] 是否避免压过主结论？
