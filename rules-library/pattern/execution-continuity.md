# 连续执行规范

> 来源：用户关于“完成一步后再次发问是否继续”的纠正 | 吸收时间：2026-03-31

## 核心问题

Agent 在用户已经给出明确目标和方向后，仍把同一路径上的自然后续步骤拆成多轮请示，导致：

1. 工作流被无意义打断
2. 用户需要重复下达“继续”指令
3. 执行节奏退化成“做一步问一步”

## 核心原则

**已获方向后，默认连续执行；只有真正的不确定性才再次发问。**

## 触发条件

当满足以下条件时，必须按连续执行处理：

- 用户已经明确总体目标或接受了当前执行方向
- 后续步骤属于同一条实现路径上的自然延续
- 不需要新的高影响决策即可继续推进

## 执行规则

### 1. 默认继续

如果当前步骤完成后，存在明确的下一个自然步骤，Agent 应直接继续，而不是停下来问“要不要继续”。

典型应连续执行的场景：

- 已确认方案后，继续实现 → 验证 → 收口
- 已开始配置整理后，继续补模板、补文档、补验证
- 已进入 review/fix 流程后，继续修复高置信问题并验证

### 2. 允许再次发问的三类情况

只有出现以下情况之一，才应暂停并向用户确认：

- **真实 blocker**：缺少必要信息、权限、文件、环境，无法安全继续
- **高影响分叉**：接下来存在 2 个以上代价明显不同的路径，且偏好无法从上下文推断
- **目标变化**：用户原目标已被新信息推翻，继续原路径会偏离意图

### 3. 阶段完成不等于任务完成

完成某个中间阶段后，如果任务整体还未闭环，不应把“阶段完成”误判为“需要重新请求授权”。

应优先判断：

- 任务是否已经端到端完成？
- 是否还有已承诺但未落地的自然后续步骤？
- 是否还缺验证、文档、模板、清理、收口？

### 4. 回复方式

若继续执行，不要发问式停顿；应使用进度更新式表述，例如：

- “我继续把模板和 README 一并补齐。”
- “下一步我会直接验证并修正剩余不一致项。”

| [[task-centric-workflow]] | status 可记录 next step，但不代表必须暂停等待 |

## 配合方式

推荐顺序：

1. 先用 `requirements-confirmation` 解决方向与边界
2. 一旦用户确认方向，立即切换到本规范，连续执行到闭环
3. 只有出现 blocker / 高影响分叉 / 目标变化时，才回到确认流程

这两条规则应视为一前一后配合，而不是重复触发的双重请示。


## 反模式

| 反模式 | 正确做法 |
|--------|----------|
| 完成一步就问“要不要继续” | 判断是否存在自然后续步骤，若有则直接继续 |
| 把阶段完成当成新的决策点 | 先判断是否只是同一路径上的连续推进 |
| 没有 blocker 也停下来请示 | 无 blocker 时默认执行到闭环 |
| 用户已说“继续”后仍频繁再次确认 | 将“继续”视为对当前执行路径的持续授权 |

## 检查清单

- [ ] 用户是否已明确目标或接受当前方向？
- [ ] 下一个步骤是否属于自然延续？
- [ ] 是否真的出现 blocker / 高影响分叉 / 目标变化？
- [ ] 若没有，是否应直接继续而不是再次发问？
