# 通用/全栈领域架构阶段定义

> 来源：从前端 `ui-logic-boundary.md` 验证的三阶段模式抽象，配合 `rules-library/core/architecture-evolution.md` 核心协议 | 吸收时间：2026-04-14

## 适用范围

本文件定义了**不属于特定领域**的通用项目阶段描述与升级触发器，适用于：

- CLI 工具 / 命令行脚本项目
- 全栈 Web 项目（前后端在同一仓库）
- 库 / SDK 项目
- Agent / 自动化 pipeline 项目
- 无法清晰归类为纯前端 / 纯后端 / ML / 桌面原生的混合项目

**优先级说明**：当项目可明确归类为特定领域时，优先使用对应的领域特定规范：

| 领域 | 优先使用 |
|------|---------|
| 前端（React / Vue 等） | `rules/domain/frontend/ui-logic-boundary.md` |
| 后端 Service | `rules/domain/backend/architecture-stages.md` |
| Rust + egui 桌面 | `rules/domain/native/rust-egui-testing.md` |
| ML 训练代码 | `rules/domain/ml/ml-training-preflight-checks.md` |
| 以上都不是 | **本文件** |

## 三阶段定义

### Stage 1: Bootstrap

```yaml
目标: 快速闭环，验证核心功能可行，避免过早架构化

适用状态:
  - 单文件或 2-3 个文件起步
  - 核心功能尚未稳定
  - 未出现跨模块复用需求

允许的简化:
  - 单文件兼任入口、逻辑、I/O
  - 配置可以硬编码或用简单常量集中存放
  - 测试可以是手工验证或极少量脚本

必须保留的边界:
  - I/O（文件读写、网络请求、系统调用）必须至少通过函数边界与核心逻辑分离
  - 不允许业务逻辑与外部副作用完全内联混合
  - 入口点（main / entrypoint / handler）应可识别，不与业务逻辑完全合并

典型形态:
  - 单个 main.py / index.ts / main.rs
  - 少量 helper 函数放在同一文件末尾
  - README 描述用途和运行方法
```

### Stage 2: Growth

```yaml
目标: 从"能跑"过渡到"可维护"，减少单文件职责

适用状态:
  - 出现第二个相似功能模块
  - 单文件开始持有多个独立职责
  - 同类逻辑在 2+ 个地方重复

必须新增:
  - 独立的模块文件（按职责拆分，而不是按大小拆分）
  - 公共接口 / 导出的明确定义
  - 配置/常量独立于逻辑文件

允许保留:
  - 少量本地临时状态与逻辑放在调用点附近
  - 简单的内联格式化和转换（不需要复用的）

重构原则:
  - 离开当前调用点后仍有意义的逻辑 → 应拆出为独立模块
  - 只服务单一入口的格式化、展示 → 可留原处
  - 新拆出的文件应让其他成员一眼看懂它的存在理由
```

### Stage 3: Structured

```yaml
目标: 完整分层，多模块协作，核心逻辑可独立测试

适用状态:
  - 多个入口或多条执行路径
  - 跨模块联动复杂，回归风险明显
  - 需要在不启动完整 runtime 的情况下验证核心行为

必须满足:
  - 明确的入口层 → 业务逻辑层 → 数据/IO 层三层分离
  - 核心业务逻辑可不依赖外部 I/O 直接引用和测试
  - 公共 contract（接口 / 类型 / schema）集中定义，面向消费方而非内部实现
  - 配置管理独立，支持不同环境注入

典型形态:
  - src/
  -   cli/ 或 api/          # 入口层
  -   core/ 或 domain/      # 业务逻辑层（纯函数/纯类，无副作用）
  -   adapters/ 或 io/      # I/O 与外部系统适配
  -   config/               # 配置与常量
  -   types/ 或 contracts/  # 类型定义
  - tests/                  # 测试（镜像 src 结构）
```

## 通用升级触发器（含量化阈值）

以下任一指标命中时，必须评估是否需要从当前阶段升级：

### 量化触发器

| 触发器 | 阈值 | 说明 | 目标阶段 |
|--------|------|------|----------|
| 单文件行数 | > 200 行且仍在增长 | 超出单次上下文可维护范围 | 至少 Stage 2 |
| 单文件函数/类数量 | > 15 个 | 单文件职责过多 | 至少 Stage 2 |
| 重复代码模式 | >= 2 处出现 | 复用需求已出现，应抽到公共模块 | 至少 Stage 2 |
| 单文件外部模块 import 数量 | > 10 个不同外部模块 | 依赖过多意味着职责过宽 | 至少 Stage 2 |
| 目录深度失衡 | 某一目录 > 10 文件，而兄弟目录为空 | 功能集中在单一位置，未按职责分布 | 至少 Stage 2 |
| 配置与业务逻辑混合 | 任何时候硬编码配置出现在 > 2 个业务文件 | 配置散落，变更成本高 | 至少 Stage 2 |
| 跨模块循环依赖 | 出现任意环形依赖 | 架构边界已经崩溃 | Stage 3（重构边界） |
| 核心行为测试依赖完整 runtime | 任何时候 | 说明业务逻辑与 I/O 未分离 | Stage 3 |
| 同一 bug 修复需要改动文件数 | > 3 个不相关文件 | 职责边界模糊 | Stage 3 |

### 模式触发器

以下模式命中时，同样需要评估升级：

- 某段逻辑需要单独被测试、复用，或被频繁修改
- 新增功能需要跨 3+ 个现有模块协调
- 文件开始管理 retry / reconnect / rollback 之类的状态机
- 同一变更散布在多个文件而没有统一入口管理
- 团队成员开始需要"问一下这个逻辑在哪里"

## 不同项目类型的阶段映射参考

| 项目类型 | Bootstrap 典型形态 | Growth 触发时机 | Structured 必要条件 |
|----------|-------------------|----------------|-------------------|
| CLI 工具 | 单 main 文件 + 几个辅助函数 | 子命令超过 3 个，或参数解析逻辑超过 50 行 | 子命令独立模块 + 纯逻辑层可单测 |
| 脚本项目 | 单脚本顺序执行 | 脚本开始被其他脚本调用，或出现重复逻辑 | 工具函数库独立 + 入口脚本只做编排 |
| 全栈 Web | 单 server 文件 + 简单路由 | 路由超过 5 个，或业务逻辑混在路由处理器里 | 路由 / service / repo 三层分离 |
| 库 / SDK | 单入口文件导出所有内容 | 导出项超过 10 个，或内部模块开始相互依赖 | 明确的 public API + 内部实现分离 |
| Agent pipeline | 单链路脚本 | 出现条件分支或多步骤复用 | 步骤模块化 + prompt/工具/执行解耦 |

## 决策框架

```text
项目处于哪个阶段？
    │
    ├─ 文件 <= 3，单一功能，无复用需求 → Stage 1: Bootstrap
    │
    ├─ 出现量化触发器（单文件 > 200 行 / import > 10 / 重复 >= 2 处）?
    │       │
    │       ├─ 是 → 升级到 Stage 2: Growth
    │       │         → 按职责拆出独立模块
    │       │         → 配置独立存放
    │       │
    │       └─ 否 → 继续 Stage 1
    │
    ├─ 出现循环依赖 / 测试需完整 runtime / 跨 3+ 模块协调?
    │       │
    │       ├─ 是 → 升级到 Stage 3: Structured
    │       │         → 建立入口/业务/IO 三层
    │       │         → 核心逻辑从 I/O 中解耦
    │       │
    │       └─ 否 → 继续 Stage 2
    │
    └─ 归属特定领域（前端/后端/ML/桌面）?
            → 优先使用对应领域的规范文件
```

## 升级执行约束

升级操作必须同时满足以下条件（与 `architecture-evolution.md` 对齐）：

- 触发器确实命中（不是主观感觉"该重构了"）
- 单次升级范围 < 5 文件（超出则拆批次，记为 Incidental finding）
- 不引入项目中不存在的全新分层
- 升级后原有行为不变（需回归验证）

## 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 项目初始就建完整分层目录 | over-engineering，agent 会忠实填充不必要结构 | Start Minimal，只建入口约定和 I/O 函数分离 |
| 触发器未命中就"顺手重构" | 变成无边界的持续清理 | 无触发不升级，记为 Incidental finding |
| 一次升级拆 10+ 文件 | 超出 agent 安全操作范围，回归风险高 | 拆成多次 < 5 文件的小手术 |
| 用"以后会用到"为理由提前建抽象 | 预判可能错误，抽象方向需等使用场景验证 | 等重复出现触发器后再抽 |
| 看到大文件就无脑拆成很多小文件 | 脱离项目语义的碎片化，找不到逻辑入口 | 只抽可复用、可测试、边界清晰的逻辑 |
| Stage 2 后继续把所有新逻辑堆回同一文件 | 触发器重复命中，技术债持续累积 | 升级后保持新代码落到对应层 |
| 配置与逻辑混合但以"方便"为由保留 | 配置变更时需要找散落在多处的硬编码 | 配置独立存放，业务代码通过参数或注入消费 |
| 测试只能通过启动完整 runtime 运行 | 核心逻辑与 I/O 未分离，定位慢 | 核心逻辑抽为纯函数，直接 import 单测 |
