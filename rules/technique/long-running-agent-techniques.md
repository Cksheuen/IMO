# 长时运行 Agent 技术规范

> 来源：[Anthropic Engineering](https://www.anthropic.com/engineering/harness-design-long-running-apps)

## Context Anxiety 处理规范

### 识别信号

当 Agent 出现以下行为时，**立即**检查 Context Anxiety：

- [ ] 突然说"差不多完成了"但明显还有工作
- [ ] 开始匆忙收尾，跳过细节
- [ ] 输出质量突然下降
- [ ] 上下文使用超过 70%

### 处理流程

```
检测到 Anxiety 信号
    │
    ▼
┌─────────────────────────┐
│ 生成 handoff.md         │
│ - 当前进度              │
│ - 未完成任务            │
│ - 关键决策              │
│ - 下一步行动            │
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│ Context Reset           │
│ - 清空上下文            │
│ - 启动新 Session        │
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│ 新 Session 读取 handoff │
│ 继续工作                │
└─────────────────────────┘
```

### Handoff 文档格式

```markdown
# Handoff - [任务名称]

## 进度
- [x] 已完成项 1
- [x] 已完成项 2
- [ ] 进行中: [当前工作]

## 关键决策
1. [决策] - [原因]

## 已知问题
- [问题描述]

## 下一步
1. [具体行动]
```

### 模型差异

| 模型 | Context Anxiety | 建议 |
|------|-----------------|------|
| Sonnet 4.5 | 强 | 必须 Context Reset |
| Opus 4.5 | 中 | 按需 Reset |
| Opus 4.6 | 弱 | 优先 Compaction，必要时 Reset |

## 评估标准设计规范

### 设计流程（必须遵循）

```
1. 识别评判维度
   └─ 问：什么是"好"的关键要素？

2. 明确定义
   └─ 问：这个维度具体指什么？

3. 设定权重
   └─ 问：哪些更重要？

4. 提供示例
   └─ 问：高分/低分长什么样？

5. 设定阈值
   └─ 问：多少分算通过/失败？
```

### 标准格式

每个评估维度**必须**包含：

```markdown
## [维度名称]

| 属性 | 值 |
|------|-----|
| 权重 | 高/中/低 |
| 通过阈值 | X/10 |
| 失败阈值 | < Y/10 |

### 定义
[一句话定义]

### 高分标准
- [具体表现]

### 低分标准
- [具体表现]

### 反例（AI 默认行为）
- [常见 AI slop 模式]
```

### Few-Shot 校准（可选但推荐）

提供 1-3 个带分数分解的示例：

```markdown
### 示例：设计评分 8/10

| 维度 | 分数 | 理由 |
|------|------|------|
| Design | 9 | 深色主题连贯，有明确技术感 |
| Originality | 8 | 非标准布局，部分图标偏通用 |
| Craft | 7 | 排版良好，按钮间距略有不一致 |
| Functionality | 8 | 导航清晰，主要流程顺畅 |
```

## Sprint Contract 规范

### 使用条件

**必须使用**：
- 任务 > 1 小时
- 多人协作（或 Agent 协作）
- 规格高层的

**可跳过**：
- 单一明确任务
- 执行时间 < 30 min

### Contract 内容要求

```markdown
## Sprint [N]: [名称]

### 范围（必须）
- [ ] [要做的 1]
- [ ] [要做的 2]

### 明确排除（必须）
- [这次不做的]

### 验收标准（必须，3-7 条）
1. [可测试的行为]
   - 验证方法: [如何测试]
2. [可测试的行为]
   - 验证方法: [如何测试]

### 风险（可选）
- [可能的问题]
```

### Contract 协议

```
Generator 提议 Contract
    │
    ▼
Evaluator 审核是否合理
    │
    ├─ 通过 → 签署 Contract → 开始编码
    │
    └─ 问题 → 反馈 → Generator 修订 → 重新审核
```
