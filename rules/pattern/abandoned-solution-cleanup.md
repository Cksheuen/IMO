# 废弃方案清理规范

> 来源：用户关于“方案切换后旧实现残留导致长期冗余”的问题抽象 | 吸收时间：2026-04-10

## 触发条件

当执行过程中出现以下任一情况时，必须启用本规范：

- 用户明确否定、撤回、替换先前批准过的方案
- 执行中确认先前方案是错误路径，且后续已切换到另一条实现路径
- 同一目标出现 A/B 两套实现，而其中一套已经确认不会继续
- 早先为旧方案新增的代码、配置、文档、测试、脚本已经不再服务当前方案

## 核心原则

**一旦旧方案确认废弃，就不能让它继续作为“隐形库存”留在项目里。**

| 原则 | 含义 |
|------|------|
| **Single Active Path** | 同一目标默认只保留一条当前有效的实现路径 |
| **Reject Means Cleanup** | 用户否定旧方案，不只是“停止继续做”，还包括清理已失效产物 |
| **Overlap Can Stay, Dead Branch Must Go** | 与新方案共享且仍有价值的部分可以保留；只服务旧方案的内容必须删除 |
| **Cleanup In Same Loop** | 一旦确认切方案，应在当前执行闭环内完成清理，而不是留待未来 |

## 执行流程

### Step 1: 明确“旧方案已废弃”这一事实

先把当前状态收敛成一句清晰判断：

```markdown
## 方案状态

- 当前批准方案：B
- 已废弃方案：A
- A 与 B 的共享部分：...
- A 的专属残留：...
```

只有在“旧方案已无继续价值”这个判断成立后，才进入自动清理。

### Step 2: 盘点旧方案产物

对旧方案留下的内容做最小盘点，至少覆盖：

- 代码实现：文件、函数、类、分支、特性开关、兼容层
- 配置与脚本：env、开关、构建逻辑、迁移脚本、临时命令
- 测试：只验证旧方案行为的测试、fixture、mock
- 文档与任务状态：spec、status、注释、README、todo

### Step 3: 按“是否仍服务当前方案”分类

对每个残留项，按下面四类处理：

| 分类 | 处理方式 |
|------|----------|
| 仍然被当前方案直接使用 | 保留 |
| 可复用于当前方案，但需要改名/改接线 | 纳入当前方案并完成重接 |
| 只服务旧方案 | 直接删除 |
| 是否仍有价值无法判断 | 先确认，不要以“以后可能有用”为由默认保留 |

判断标准只有一个：**删掉它后，当前已批准方案是否仍完整、正确、可验证。**

### Step 4: 在同一 diff 中完成“切方案 + 清残留”

当新方案开始落地时，旧方案残留应作为同一次交付的一部分被清理：

- 删除旧入口、旧分支、旧特性开关
- 删除只服务旧方案的测试、fixture、脚本、文档描述
- 更新 status/spec/注释，避免文字层面仍描述旧路径
- 清理因旧方案引入但现在已无消费者的抽象层

禁止只新增 B 而把 A 留在仓库里“以后再说”。

### Step 5: 收尾前做残留审查

交付前至少执行一次“废弃方案残留检查”：

- 是否仍存在旧方案命名、旧开关、旧注释、旧 TODO
- 是否还有永远不会被走到的分支或无人使用的兼容层
- 测试与文档是否仍在描述已被否定的行为
- 当前 diff 是否同时体现了“新方案落地”和“旧方案回收”

## 决策框架

```
发现先前方案被否定？
    │
    ├─ 否 → 不触发本规范
    │
    └─ 是 → 旧内容是否仍服务当前批准方案？
            │
            ├─ 是 → 保留或重接到当前方案
            │
            ├─ 否 → 删除
            │
            └─ 不确定 → 先确认，禁止默认保留
```

## 与现有规则的关系

| 规则 | 关系 |
|------|------|
| `rules/pattern/requirements-confirmation.md` | 负责减少一开始走错方向；本规范负责已经走错或已切方向后的回收 |
| `rules/pattern/living-spec.md` | 负责记录方向变化；本规范负责把旧方向的代码和文档残留真正清掉 |
| `rules/pattern/change-scope-guard.md` | 负责防止顺手扩范围；本规范要求对已确认废弃的旧内容做必要清理，这属于当前任务范围的一部分 |
| `rules/pattern/change-impact-review.md` | 清理旧方案后仍需做回归验证，避免删掉共享依赖 |

## 反模式

| 反模式 | 正确做法 |
|--------|----------|
| 用户否定 A 后，只新增 B，不删除 A | 在同一轮实现 B 时清理 A 的残留 |
| 以“以后可能有用”为理由保留死代码 | 不确定就确认；确认无用后直接删除 |
| 只删代码，不更新 spec/status/注释 | 代码和文档状态必须同步 |
| 保留旧开关、旧分支、旧测试，假装以后再收 | 切方案时一并回收，不制造技术债挂账 |
| 误删 A/B 共享部分导致回归 | 先分类共享与专属，再删除，并做回归验证 |

## 检查清单

- [ ] 是否已经明确哪个方案是当前批准方案，哪个方案已废弃？
- [ ] 是否盘点了旧方案留下的代码、配置、测试、文档产物？
- [ ] 是否只保留了仍服务当前方案的共享部分？
- [ ] 是否删除了只服务旧方案的实现、测试、脚本、文档描述？
- [ ] 是否更新了 spec/status/注释，避免继续描述旧路径？
- [ ] 是否验证清理后当前方案仍完整可用？
