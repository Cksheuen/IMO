# 项目架构优先规范

> 来源：围绕“最小模块化 + 项目架构一致性 + CC/Codex 共享约束”的收敛设计 | 吸收时间：2026-04-13

## 核心问题

当 Agent 带着个人偏好进入新项目时，最容易犯的错不是代码写不出来，而是：

- 没先理解项目当前架构就开始改
- 为了“更优雅”引入项目里不存在的新分层或新目录
- 把“最小模块化”误写成“过度原子化”或“继续把所有逻辑塞在大文件里”
- 多人协作时，每个人都按自己的结构习惯扩展，导致仓库越来越不一致

## 核心原则

**项目现有架构优先于 Agent 的个人默认偏好。**

先读清楚，再修改；先沿现有结构扩展，再讨论是否应该重构结构。

## 触发条件

当出现以下任一情况时，必须应用本规范：

- 第一次进入某个项目或第一次改某个陌生模块
- 用户要求实现 / 修复 / 重构 / 写测试 / 评审代码
- 需要决定文件该放哪、逻辑该拆到哪一层、模块边界该怎么划
- 需要提升可测试性、模块化或多人协作一致性

## 执行规范

### 1. 先做项目架构预读，再写代码

在真正修改前，优先读取下列入口中的高信号文件：

- 项目指令入口：`AGENTS.md`、`CLAUDE.md`
- 项目说明入口：`README.md`
- 架构 / 设计入口：`docs/architecture*.md`、`docs/design*.md`、`notes/design/`
- 技术栈入口：`package.json`、`pyproject.toml`、`Cargo.toml`、`go.mod`
- 当前任务入口：`.claude/tasks/current/context.md`、`prd.md`

若这些入口存在，就不能跳过它们直接凭经验改代码。

### 2. 先提取“当前项目的真实结构事实”

至少确认以下事实后，才进入实现：

- 主要技术栈与运行时边界
- 目录分层：页面 / 组件 / hooks / services / adapters / runtime / tests 在哪里
- 现有模块粒度：该项目偏大文件、页面域拆分，还是严格分层
- 命名与落盘习惯：新逻辑通常放在哪类文件
- 测试入口与测试镜像结构

若这些事实还不清楚，继续读文件；不要靠猜。

### 3. 模块化必须“顺着项目架构拆”，不是脱离上下文地拆

当为了可测试性或可维护性做模块化时：

- 不允许继续把多个可复用流程堆在同一个大文件里
- 也不允许为了“最小模块化”强行制造项目里没有的细碎层级
- 优先复用项目已有层名和目录语义
- 优先抽离“可复用、可测试、可复审”的逻辑边界

判断标准：

- 离开当前 UI / 当前调用点后仍有意义的逻辑 -> 应拆出
- 只服务当前视觉表现的轻量状态或格式化 -> 可留原处
- 新拆出的文件应能被项目其他成员一眼看懂它为什么放在这里

### 4. 架构一致性优先于个人偏好

如果项目当前已经存在稳定模式：

- 新代码默认沿用该模式
- 不因外部最佳实践就直接重塑目录
- 不因个人偏好就引入第二套并行架构

只有在以下情况，才允许主动推动结构升级：

- 当前结构已经明确阻塞本任务
- 同一问题反复出现，且根因就是结构失衡
- 用户明确要求做架构重构

### 5. 不熟悉项目时，先输出架构判断，再动手

若当前任务涉及跨模块修改或结构调整，至少应先形成以下判断：

- 当前项目属于什么分层风格
- 本次改动应落在哪一层
- 明确不引入哪些新层 / 新目录 / 新模式

## 决策框架

```text
进入新项目 / 新模块？
    │
    ├─ 是 → 先读架构入口文件
    │       → 提取当前分层、目录、命名、测试入口
    │       → 再决定如何修改
    │
    └─ 否 → 沿用已确认的项目结构继续

需要模块化？
    │
    ├─ 逻辑可复用 / 可测试 / 跨页面有意义
    │       → 抽出，且落到项目已有层级
    │
    ├─ 只服务当前 UI 表现
    │       → 可留在原文件
    │
    └─ 不确定
            → 优先问：这个逻辑离开当前文件后是否仍成立？
               若成立，则不应继续塞在大文件里
```

## 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 不读项目结构，直接按自己习惯开新目录 | 架构漂移 | 先读入口文件，再按现有结构扩展 |
| 看到大文件就无脑拆成很多 utils | 脱离项目语义的碎片化 | 只抽可复用、可测试、边界清晰的逻辑 |
| 为了“最佳实践”引入项目里不存在的新层 | 多人协作成本上升 | 优先复用当前项目已有分层 |
| 为了赶进度继续把所有逻辑留在一个文件里 | 长期失控 | 识别稳定可复用逻辑后及时抽离 |
| 架构不清楚时边改边猜 | 易引入错误结构 | 先做架构预读，后实现 |
