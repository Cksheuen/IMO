# 用户交流语言规范

> 来源：当前配置中“持久化内容默认中文”覆盖了文档层，但未稳定约束用户交互输出语言 | 吸收时间：2026-04-10

## 核心问题

Agent 当前对“中文”约束主要落在 `skills/`、`rules/`、`notes/`、`tasks/` 等持久化内容，导致用户交互层仍可能混入英文默认话术、英文状态更新或英文结论。

## 核心原则

**只要是在和用户说话，默认使用中文。**

代码、命令、路径、报错原文、协议字段名可以保留英文，但包裹它们的解释、判断、结论和行动说明应使用中文。

## 触发条件

当出现以下任一情况时，必须应用本规范：

- 向用户发送进度更新
- 向用户提问、确认、说明 blocker
- 输出最终结论、review 结果、测试结果
- 解释命令、日志、报错、配置项含义
- 生成任何直接面向用户阅读的操作建议

## 执行规范

### 1. 用户可见文本默认中文

以下内容默认必须使用中文：

- commentary / intermediary updates
- final answer / 结果说明
- review findings 的标题、结论、风险说明
- 测试通过/失败说明
- 权限申请、下一步动作说明

### 2. 英文只保留在对象层

以下内容可以保留英文原样：

- 代码、命令、路径、环境变量
- API 名称、字段名、类型名
- 原始报错、日志片段
- 用户明确要求保留英文的原文引用

但这些英文对象前后的解释文本，仍应使用中文。

### 3. 切换语言必须有明确依据

只有以下情况可以整体改用英文或中英混排：

- 用户明确要求英文回答
- 正在产出必须为英文的文档或文案
- 若翻译会引入歧义，需保留英文术语并用中文解释

若无明确依据，不要因为工具默认提示词、外部 skill 描述或上游英文模板而切换成英文。

## 反模式

| 反模式 | 正确做法 |
|--------|----------|
| 用户用中文提问，Agent 用英文 progress update 回复 | progress update 用中文，命令本体保留英文 |
| 结论用英文，只有代码片段是英文 | 结论用中文，代码片段单独保留英文 |
| 因为 system/developer prompt 是英文，就沿用英文对用户输出 | 区分内部指令语言与用户可见输出语言 |
| 为了“专业”把 review findings 标题全部写成英文 | findings 标题与说明用中文，文件路径/符号名保留英文 |
