# Eat - 知识吸收元技能

吸收一切知识，转化为持久能力。

## 核心理念

你是一个持续进化的系统。每次知识吸收都是一次能力增强：
- 看到好的 X 帖子 → 提取思维框架
- 发现优秀的 GitHub 项目 → 学习架构设计
- 读到精妙的技术文章 → 内化实现技巧

这些知识会沉淀到 `rules/` 目录或转化为独立的 `skills/`，让你在未来遇到类似场景时自动应用。

## 输入类型

| 类型 | 获取方法 | 特点 |
|------|----------|------|
| URL（X/GitHub/博客） | WebFetch / WebSearch | 需要网络请求 |
| 本地文件 | Read | 直接读取 |
| 直接粘贴 | 无需工具 | 已在上下文中 |
| 多媒体（图片/PDF） | OCR / 提取工具 | 需要解析 |

## 知识分类体系

| 类型 | 目录 | 特征 | 示例 |
|------|------|------|------|
| **technique** | `rules/technique/` | 具体实现技巧 | 代码模式、性能优化 |
| **pattern** | `rules/pattern/` | 架构模式 | 设计模式、系统架构 |
| **best-practice** | `rules/best-practice/` | 推荐做法 | 错误处理、安全实践 |
| **anti-pattern** | `rules/anti-pattern/` | 反模式警告 | 常见陷阱 |
| **tool** | `rules/tool/` | 工具使用 | CLI、MCP、库 |
| **workflow** | `skills/` | 可独立运行 | 完整工作流 |

## Phase 1: 内容获取与预处理

```
输入识别
    │
    ├─ URL → WebFetch / WebSearch / mcp__exa__web_search_exa
    │
    ├─ 本地文件 → Read
    │
    ├─ 直接粘贴 → 无需工具
    │
    └─ 多媒体 → OCR / 提取工具
```

**输出**：获取原始内容，进入 Phase 2 分析。

---

## Phase 2: 深度分析与提取

### 核心问题（必须回答）

1. **核心价值是什么？** - 这内容最有价值的是什么？
2. **可迁移的 Pattern 是什么？** - 什么可以应用到其他场景？
3. **与现有知识的关系？** - 是否已存在？如何关联？
4. **输出类型？** - 应该是知识、规范还是工作流？

### 分析维度

| 维度 | 关注点 | 提取目标 |
|------|--------|----------|
| **架构层面** | 设计理念、核心模块关系 | 系统结构 pattern |
| **实现层面** | 具体技术、代码技巧 | 可复用代码模式 |
| **决策层面** | 关键 trade-off、选择理由 | 决策框架 |

### 工具/技能吸收判断

当内容提到具体的 **skill、MCP、工具、库** 时：

```
提到具体工具/skill/MCP？
    │
    ├─ 否 → 跳过此步骤
    │
    └─ 是 → 能否直接使用？
            │
            ├─ 能（npm/MCP配置/远程调用）→ 不拉取，记录使用方法
            │
            └─ 不能（需定制/私有部署）→ 拉取到本地，记录原因
```

| 条件 | 处理 |
|------|------|
| npm 包可直接安装 | 不拉取，记录安装命令 |
| MCP 可远程配置 | 不拉取，记录配置方法 |
| Skill 可直接使用 | 不拉取，记录调用方式 |
| 需定制化修改 | 拉取到本地 skills/ |

**输出**：完成分析，进入 Phase 3 写入判断。

---

## Phase 3: 知识写入

### 输出类型判断

```
内容性质？
    │
    ├─ 知识（参考性、理解性）→ rules/knowledge/
    │
    ├─ 规范（可执行、有触发条件）→ rules/pattern/ 或 rules/technique/
    │
    └─ 工作流（可独立运行）→ skills/
```

**判断问题**：
1. 这个内容是用来"做决策/执行"还是"参考/理解"？
2. 未来遇到类似场景时，如何自动触发？
3. 用户需要"知道"还是"做到"？

### 去重检查（写入前必须执行）

**检查范围**：
- `~/.claude/rules/**/*.md`
- `~/.claude/projects/**/memory/*.md`
- `~/.claude/skills/*/SKILL.md`

**检查方法**：
```bash
# 关键词搜索
Grep pattern="[关键词]" path="~/.claude/rules"
Grep pattern="[关键词]" path="~/.claude/projects"

# 文件名匹配
Glob pattern="~/.claude/rules/**/*[关键词]*"
```

### 处理策略

| 发现情况 | 处理 |
|---------|------|
| 完全重复 | 跳过，提示用户已存在 |
| 部分重叠 | 合并精华，补充新内容 |
| 主题相关但不同 | 新建文件，建立链接 |
| 与 CLAUDE.md 冲突 | 对比优劣，更新或放弃 |

**输出**：确定写入位置和方式，进入 Phase 4 验证。

---

## Phase 4: 验证与更新

### 规范验证（输出为规范时）

**必须包含**：
- [ ] **触发条件** - 什么情况下应用
- [ ] **执行步骤** - 具体怎么做
- [ ] **决策框架** - 如何判断/选择
- [ ] **相关规范** - 与其他规范的链接

### 知识验证（输出为知识时）

**必须包含**：
- [ ] **核心洞察** - 一句话概括价值
- [ ] **详细内容** - 具体知识点
- [ ] **应用场景** - 何时使用

### 更新索引

写入后更新相关索引：
- 重要规则 → 更新 `MEMORY.md` 索引
- 相关规则 → 在"相关规范"中建立双向链接

## 关键原则

1. **提取而非复制** - 不要简单存储原文，要提取可迁移的 pattern
2. **原子化** - 每个 rule 聚焦单一概念，便于组合
3. **可检索** - 命名清晰，便于未来查找和触发
4. **增量积累** - 新知识与现有规则建立关联，形成知识网络

## 与其他技能的协作

- **工作流类内容** → 调用 `/skill-creator` 创建新 skill
- **归档旧知识** → 调用 `/archive` 移到冷存储
