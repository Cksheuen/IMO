# Eat - 知识吸收元技能

吸收一切知识，转化为持久能力。

## 核心理念

你是一个持续进化的系统。每次知识吸收都是一次能力增强：
- 看到好的 X 帖子 → 提取思维框架
- 发现优秀的 GitHub 项目 → 学习架构设计
- 读到精妙的技术文章 → 内化实现技巧

这些知识会沉淀到 `rules/` 目录或转化为独立的 `skills/`，让你在未来遇到类似场景时自动应用。

## 输入类型与处理策略

### 1. X/Twitter 帖子

**处理流程：**
1. 使用 WebFetch 或 WebSearch 获取帖子内容
2. 提取核心洞察、思维框架、方法论
3. 识别可复用的 pattern

**输出示例：**
```
rules/technique/prompt-chaining.md
```

### 2. GitHub 项目

**处理流程：**
1. 获取项目 README 和核心文件结构
2. 分析架构设计理念
3. 提取关键实现技巧
4. 识别可复用的设计模式

**输出示例：**
```
rules/pattern/plugin-architecture.md
rules/technique/react-performance-optimization.md
```

### 3. 技术文章/博客

**处理流程：**
1. 提取文章核心观点
2. 识别关键技术决策
3. 总结可复用的最佳实践

**输出示例：**
```
rules/best-practice/error-handling-strategy.md
```

### 4. 代码片段

**处理流程：**
1. 理解代码意图
2. 提取设计思想
3. 识别可泛化的技巧

**输出示例：**
```
rules/technique/elegant-recursion.md
```

### 5. 工作流程

**处理流程：**
1. 识别工作流的关键步骤
2. 评估是否可泛化为 skill
3. 如适合，使用 `/skill-creator` 创建新 skill

**输出示例：**
```
skills/new-workflow-skill/SKILL.md
```

## 知识分类体系

按类型组织，存入对应目录：

| 类型 | 目录 | 示例 |
|------|------|------|
| **technique** | `rules/technique/` | 具体实现技巧、代码模式 |
| **pattern** | `rules/pattern/` | 架构模式、设计模式 |
| **best-practice** | `rules/best-practice/` | 最佳实践、推荐做法 |
| **anti-pattern** | `rules/anti-pattern/` | 反模式、常见陷阱 |
| **tool** | `rules/tool/` | 工具使用技巧 |
| **workflow** | `skills/` | 可独立运行的技能 |

## 工具/技能吸收判断（关键步骤）

当文章提到具体的 **skill、MCP、工具、库** 时，必须判断是否需要拉取到本地。

### 判断决策树

```
文章提到了具体的工具/skill/MCP？
    │
    ├─ 否 → 跳过此步骤
    │
    └─ 是 → 能否直接使用？
            │
            ├─ 能（npm install / MCP 配置 / 远程调用）→ 不拉取本地
            │                                            只记录使用方法
            │
            └─ 不能（需要定制化 / 本地修改 / 私有部署）→ 拉取到本地
                                                          并记录修改原因
```

### 判断标准

| 条件 | 处理方式 | 示例 |
|------|----------|------|
| **npm 包可直接安装** | 不拉取，记录安装命令 | `npm i -g dev-browser` |
| **MCP 可远程配置** | 不拉取，记录配置方法 | Playwright MCP |
| **Skill 可直接使用** | 不拉取，记录调用方式 | `/skill-name` |
| **需要定制化修改** | 拉取到本地 skills/ | 修改后的工作流 skill |
| **需要私有部署** | 拉取到本地 | 内部 MCP server |

### 输出格式

**直接使用的工具**（在知识文件中记录）：

```markdown
## 相关工具

- **dev-browser**: `npm i -g dev-browser` - 浏览器自动化 CLI
- **Playwright MCP**: 在 Claude 配置中添加 MCP server
```

**需要拉取的工具**（实际复制文件）：

```markdown
## 本地化工具

- `skills/custom-browser/` - 基于 dev-browser 定制，增加了 [功能]
```

## 去重检查（关键步骤）

**在写入前必须执行**，避免冗余和冲突。

### 检查范围

```
~/.claude/
├── rules/**/*.md      # 所有规则
├── projects/**/memory/*.md  # 记忆文件
├── skills/*/SKILL.md  # 所有技能
└── CLAUDE.md          # 核心配置
```

### 检查方法

1. **关键词搜索**：使用 Grep 搜索相关主题词
   ```bash
   # 示例：搜索 "worktree" 相关内容
   Grep pattern="worktree" path="~/.claude/rules"
   Grep pattern="worktree" path="~/.claude/projects"
   ```

2. **文件名匹配**：使用 Glob 检查是否有同名或相似文件
   ```bash
   Glob pattern="~/.claude/rules/**/*worktree*"
   ```

3. **语义相似度**：对比新内容与现有内容的核心概念

### 处理策略

| 发现情况 | 处理方式 |
|---------|---------|
| **完全重复** | 跳过，提示用户已存在 |
| **部分重叠** | 合并精华到现有文件，补充新内容 |
| **主题相关但不同** | 新建文件，在"相关规则"中建立链接 |
| **与 CLAUDE.md 冲突** | 对比优劣，更新 CLAUDE.md 或放弃 |

### 合并原则

1. **取其精华**：保留两边的核心价值
2. **避免冗余**：删除重复描述
3. **保持简洁**：合并后文件不应过长
4. **更新时间戳**：标注"更新自 [来源]"和更新日期

## 输出类型判断（关键步骤）

**吸收前必须判断输出类型**，选择对应模板：

| 类型 | 特征 | 存储位置 | 模板 |
|------|------|----------|------|
| **知识** | 描述性、参考性、理解性 | `rules/knowledge/` | 核心洞察 → 内容 → 场景 |
| **规范** | 可执行、有触发条件、有决策 | `rules/pattern/`, `rules/technique/` | 触发条件 → 执行步骤 → 决策框架 |

**判断问题**：
1. 这个内容是用来"做决策/执行"还是"参考/理解"？
2. 未来遇到类似场景时，如何自动触发？
3. 用户需要"知道"还是"做到"？

**判断结果 → 模板选择**：
- 答案是"执行/做到" → 使用**规范模板**
- 答案是"理解/知道" → 使用**知识模板**

## 输出格式

### 规范文件模板（可执行）

用于：pattern、technique、best-practice

```markdown
# [标题]

> 来源：[原始链接/出处]

## 触发条件

当任务满足以下条件时应用此规范：
- [条件1]
- [条件2]

## 执行规范

### 步骤

1. [具体步骤1]
2. [具体步骤2]

### 决策框架

```
[决策树或判断流程]
```

### 检查清单

- [ ] [检查项1]
- [ ] [检查项2]

## 相关规范

- [[related-rule-name]]
```

### 知识文件模板（参考性）

用于：knowledge、参考文档

```markdown
# [标题]

> 来源：[原始链接/出处]
> 吸收时间：YYYY-MM-DD

## 核心洞察

[一句话概括核心价值]

## 详细内容

[具体的知识点、模式、技巧]

## 应用场景

- 场景1：...
- 场景2：...

## 相关规则

- [[related-rule-name]]
```

## 规范验证清单

输出为规范时，**必须验证**包含以下要素：

- [ ] **触发条件** - 什么情况下应用（必须有）
- [ ] **执行步骤** - 具体怎么做（必须有）
- [ ] **决策框架** - 如何判断/选择（必须有）
- [ ] **相关规范** - 与其他规范的链接（建议有）

**验证失败**：缺少必须项 → 补充后再输出

### memory 文件模板

当知识适合放入 memory 时：

```markdown
# [主题]

[简洁的知识总结，便于上下文加载]

## 关键点

- 点1
- 点2

## 代码片段

[如有]
```

## 关键原则

1. **提取而非复制**：不要简单存储原文，要提取可迁移的 pattern
2. **原子化**：每个 rule 应该聚焦单一概念，便于组合
3. **可检索**：命名清晰，便于未来查找和触发
4. **增量积累**：新知识应该与现有规则建立关联，形成知识网络

## 与其他技能的协作

- **工作流类内容** → 调用 `/skill-creator` 创建新 skill
- **代码相关** → 写入 `rules/technique/` 或 `rules/pattern/`
- **行为规范** → 写入 `rules/` 根目录或 `MEMORY.md`
- **工具使用** → 写入 `rules/tool/`
- **归档旧知识** → 调用 `/archive` 移到冷存储
