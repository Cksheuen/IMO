# Eat - 知识吸收元技能

吸收一切知识，转化为持久能力。

## 核心理念

你是一个持续进化的系统。每次知识吸收都是一次能力增强：
- 看到好的 X 帖子 → 提取思维框架
- 发现优秀的 GitHub 项目 → 学习架构设计
- 读到精妙的技术文章 → 内化实现技巧

这些知识的目标是吸收**新资料**并形成新的能力增量。

`eat` 的职责边界是主动吸收：

- 面对新 URL、新文章、新项目、新代码片段时提取可迁移模式
- 根据新资料生成新的规则候选或 skill 候选
- 必要时把尚未稳定的新理解先写入 `notes/`

`eat` **不是** `notes` 晋升为 `rules/skills` 的默认入口；旧经验的被动升格应由独立的 `Promotion Loop` 负责。

## 输入类型

| 类型 | 获取方法 | 特点 |
|------|----------|------|
| URL（X/GitHub/博客） | WebFetch / WebSearch | 需要网络请求 |
| 本地文件 | Read | 直接读取 |
| 直接粘贴 | 无需工具 | 已在上下文中 |
| 多媒体（图片/PDF） | OCR / 提取工具 | 需要解析 |

## 知识分类体系

| 类型 | 目录 | 特征 | 示例 |
|------|------|------|------|
| **technique** | `rules/technique/` | 具体实现技巧 | 代码模式、性能优化 |
| **pattern** | `rules/pattern/` | 架构模式 | 设计模式、系统架构 |
| **best-practice** | `rules/best-practice/` | 推荐做法 | 错误处理、安全实践 |
| **anti-pattern** | `rules/anti-pattern/` | 反模式警告 | 常见陷阱 |
| **tool** | `rules/tool/` | 工具使用 | CLI、MCP、库 |
| **workflow** | `skills/` | 可独立运行 | 完整工作流 |

## Phase 1: 内容获取与预处理

```
输入识别
    │
    ├─ URL → 尝试获取
    │       │
    │       ├─ WebFetch 直接获取
    │       │       │
    │       │       ├─ 成功 → 返回内容
    │       │       │
    │       │       └─ 失败（需要鉴权）→ 鉴权获取
    │       │
    │       └─ 鉴权获取 → 参见 [[browser-auth-reuse]]
    │
    ├─ 本地文件 → Read
    │
    ├─ 直接粘贴 → 无需工具
    │
    └─ 多媒体 → OCR / 提取工具
```

### 鉴权获取流程

当 WebFetch 返回 401/403 或重定向到登录页时，应用 [[browser-auth-reuse]] 规则：

```yaml
优先级顺序:
  1. Chrome DevTools MCP 直接获取（推荐）
     - 无需处理 Cookie 传递
     - 浏览器自动携带登录态

  2. Cookie 提取 + WebFetch（备选）
     - 从 Chrome 提取 Cookie
     - 带 Cookie Header 重新请求

错误处理:
  - 用户未登录 → 提示"请在 Chrome 中登录目标网站后重试"
  - Cookie 过期 → 清除缓存，重新提取
```

**输出**：获取原始内容，进入 Phase 2 分析。

## Phase 2: 深度分析与提取

### Phase 2 前置：相关知识检索（按需执行）

> **核心原则**：`eat` 以新资料为主，但若已有相关 note/rule/skill，应先检索以避免重复吸收。

**强制动作**：

- [ ] 搜索 `notes/research/` 中是否已有同主题调研
- [ ] 如新资料明显触发已有教训，搜索 `notes/lessons/`
- [ ] 若内容涉及结构/流程设计，搜索 `notes/design/`
- [ ] 搜索已有 `rules/` 与 `skills/`，避免重复提炼
- [ ] 记录命中的知识，以及它们与当前输入的关系

**处理策略**：

| 命中情况 | 处理方式 |
|---------|---------|
| 已有相关规则/技能 | 以增量更新为主，避免重复造轮子 |
| 已有相关 note | 作为背景知识参考，但不把 `eat` 变成 note 晋升器 |
| 无相关内容 | 直接分析新资料，按结果决定写 `rules/`、`skills/` 或 `notes/` |

**证据记录**：

```text
相关知识检索记录：
- 检索关键词：...
- 命中结果：
  - notes/research/xxx.md（相关度：高/中/低）
  - notes/lessons/xxx.md（相关度：高/中/低）
  - notes/design/xxx.md（相关度：高/中/低）
  - rules/xxx.md（相关度：高/中/低）
  - skills/xxx/SKILL.md（相关度：高/中/低）
- 当前处理策略：增量吸收 / 新建规则候选 / 先记入 notes
```

### 时效性验证（必须执行）

> **核心原则**：过时的资料可能提供错误的技术建议，必须验证时效性。

当内容来自外部来源（URL、博客、论文）时，**必须**验证发布/更新时间：

| 时间范围 | 时效性评估 | 处理方式 |
|---------|-----------|---------|
| < 1 年 | ✅ 最新 | 直接采用 |
| 1-3 年 | ⚠️ 需验证 | 检查核心方法是否仍适用 |
| > 3 年 | ❌ 可能过时 | 仅参考原理，验证当前实践 |
| 数学/基础原理 | ✅ 永不过时 | 直接采用 |

**验证方法**：
```bash
# 搜索发布日期
WebSearch query="[文章标题] published date 2024 2025"

# 检查工具/库最新版本
WebSearch query="[工具名] latest version release 2024 2025"
```

**时效性评估输出**：
```
时效性验证记录：
- 来源 1: [标题] - [日期] - ✅/⚠️/❌
- 来源 2: [标题] - [日期] - ✅/⚠️/❌
- 结论：[是否可信]
```

### 核心问题（必须回答）

1. **核心价值是什么？** - 这内容最有价值的是什么？
2. **可迁移的 Pattern 是什么？** - 什么可以应用到其他场景？
3. **与现有知识的关系？** - 是否已存在？如何关联？
4. **输出类型？** - 应该是知识、规范还是工作流？

### 分析维度

| 维度 | 关注点 | 提取目标 |
|------|--------|----------|
| **架构层面** | 设计理念、核心模块关系 | 系统结构 pattern |
| **实现层面** | 具体技术、代码技巧 | 可复用代码模式 |
| **决策层面** | 关键 trade-off、选择理由 | 决策框架 |

### 工具/技能吸收判断

当内容提到具体的 **skill、MCP、工具、库** 时：

```
提到具体工具/skill/MCP？
    │
    ├─ 否 → 跳过此步骤
    │
    └─ 是 → 分析类型
            │
            ├─ MCP Server → 即时配置（见下方）
            │
            ├─ npm 包 → 即时安装（见下方）
            │
            ├─ Skill → 即时安装（见下方）
            │
            └─ 需定制/私有 → 拉取到本地 skills/
```

## Phase 2.5: 即时配置（新增）

> **核心原则**：发现工具后立即配置，让知识即刻可用。

### MCP Server 配置

当发现新的 MCP Server 时，**立即配置到 `.mcp.json`**：

```yaml
配置流程:
  1. 读取现有 .mcp.json
  2. 检查是否已存在同名配置
  3. 添加新配置到 mcpServers
  4. 提示用户重启会话

配置模板:
  {
    "mcpServers": {
      "[server-name]": {
        "command": "npx" | "bunx",
        "args": ["[package-name]", "--registry=..."],
        "env": {
          "NPM_CONFIG_REGISTRY": "...",
          // 其他环境变量
        }
      }
    }
  }
```

**示例**（飞书 MCP）：
```json
{
  "mcpServers": {
    "feishu": {
      "command": "npx",
      "args": ["@i18n-ecom/feishu-lark-mcp-server", "--registry=https://bnpm.byted.org/"],
      "env": {
        "NPM_CONFIG_REGISTRY": "http://bnpm.byted.org/"
      }
    }
  }
}
```

### npm 包安装

当发现可全局使用的 npm 包时：

| 类型 | 安装命令 |
|------|----------|
| **全局工具** | `npm install -g [package]` 或 `bun install -g [package]` |
| **项目依赖** | 提示用户在项目目录执行 `npm install [package]` |

**判断标准**：
- CLI 工具 → 全局安装
- 库/SDK → 提示项目安装

### Skill 安装

当发现可用的 skill 时：

```bash
# 使用 Skills CLI
npx skills add [owner/repo@skill] -g -y

# 或手动安装到 skills/ 目录
```

### 配置后验证

| 配置类型 | 验证方法 |
|----------|----------|
| MCP Server | 重启后调用 `ListMcpResourcesTool` |
| npm 包 | 执行 `[package] --version` |
| Skill | 执行 `/[skill-name]` |

### 用户确认策略

**默认行为**：直接配置，完成后通知用户

**需要确认的场景**：
- 安装全局 npm 包（可能影响系统环境）
- 删除/覆盖现有配置
- 不确定的配置参数

---

### 原有判断逻辑

| 条件 | 处理 |
|------|------|
| npm 包可直接安装 | 即时安装或记录命令 |
| MCP 可配置 | **即时配置到 .mcp.json** |
| Skill 可直接使用 | 即时安装或记录方法 |
| 需定制化修改 | 拉取到本地 skills/ |

**输出**：完成分析与配置，进入 Phase 3 写入判断。

## Phase 3: 知识写入

### 输出类型判断

```
内容性质？
    │
    ├─ 新资料带来新理解，但暂未稳定 → notes/
    │
    ├─ 规范（可执行、有触发条件）→ rules/pattern/ 或 rules/technique/
    │
    └─ 工作流（可独立运行）→ skills/
```

**判断问题**：
1. 这个内容是用来"做决策/执行"还是"参考/理解"？
2. 未来遇到类似场景时，如何自动触发？
3. 用户需要"知道"还是"做到"？
4. 这是来自新资料的增量理解，还是旧经验的被动升格？

### 新资料吸收后的去向判断

| 条件 | 去向 |
|------|------|
| 新资料直接形成稳定触发条件、执行步骤、决策框架 | `rules/` |
| 新资料直接形成完整工作流、步骤长、适合按需触发 | `skills/` |
| 新资料仍在探索、主要是解释材料、需要等待验证 | `notes/research/` 或 `notes/design/` |
| 新资料印证了已有 lesson，但不足以直接出规则 | 更新 `notes/lessons/` |

### 吸收前的最终判断

在写入 `rules/` 或 `skills/` 前，必须回答：

1. 这是否确实来自新资料，而不是已有 note 的升格需求？
2. 它是否已经脱离具体案例，具备可迁移性？
3. 如果现在直接写入 `rules/skills`，会不会把仍在变化的理解过早固化？

若任一答案是否定，优先回写 `notes/`。

### 去重检查（写入前必须执行）

**检查范围**：
- `~/.claude/rules/**/*.md`
- `~/.claude/projects/**/memory/*.md`
- `~/.claude/skills/*/SKILL.md`

**检查方法**：
```bash
# 关键词搜索
Grep pattern="[关键词]" path="~/.claude/rules"
Grep pattern="[关键词]" path="~/.claude/projects"

# 文件名匹配
Glob pattern="~/.claude/rules/**/*[关键词]*"
```

### 处理策略

| 发现情况 | 处理 |
|---------|------|
| 完全重复 | 跳过，提示用户已存在 |
| 部分重叠 | 合并精华，补充新内容 |
| 主题相关但不同 | 新建文件，建立链接 |
| 与 CLAUDE.md 冲突 | 对比优劣，更新或放弃 |

**输出**：确定写入位置和方式，进入 Phase 4 验证。

## Phase 4: 验证与更新

### 规范验证（输出为规范时）

**必须包含**：
- [ ] **触发条件** - 什么情况下应用
- [ ] **执行步骤** - 具体怎么做
- [ ] **决策框架** - 如何判断/选择
- [ ] **相关规范** - 与其他规范的链接

### 知识验证（输出为知识时）

**必须包含**：
- [ ] **核心洞察** - 一句话概括价值
- [ ] **详细内容** - 具体知识点
- [ ] **应用场景** - 何时使用

### 更新索引

写入后更新相关索引：
- 重要规则 → 更新 `MEMORY.md` 索引
- 相关规则 → 在"相关规范"中建立双向链接

## 关键原则

1. **提取而非复制** - 不要简单存储原文，要提取可迁移的 pattern
2. **原子化** - 每个 rule 聚焦单一概念，便于组合
3. **可检索** - 命名清晰，便于未来查找和触发
4. **增量积累** - 新知识与现有规则建立关联，形成知识网络

## 与其他技能的协作

- **工作流类内容** → 调用 `/skill-creator` 创建新 skill
- **归档旧知识** → 调用 `/archive` 移到冷存储
