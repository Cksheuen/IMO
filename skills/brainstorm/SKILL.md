# Brainstorm - 调研与需求发现

**专注调研与需求发现，不涉及任务管理。**

```
用户描述任务 → 知识库检索 → 强制调研 → 需求收敛 → 输出调研结果 → 知识归并沉淀
```

## 核心原则

| 原则 | 含义 |
|------|------|
| **Research-first** | 技术选择必须先调研业界实践，再提出方案 |
| **Action-before-asking** | 能从代码/文档/研究中推导的，不问用户 |
| **Diverge→Converge** | 先扩展思考（未来演化/边缘情况），再收敛到 MVP |
| **One-question-at-a-time** | 一次只问一个关键问题 |
| **Knowledge-reuse** | 先查已有知识，再调研新知识，最后归并沉淀新发现 |
| **Merge-before-create** | 知识沉淀优先更新同主题旧 note，而不是创建新流水账 |
| **Freshness-over-archive** | 教训类知识必须带最近验证时间，过久未验证则降级处理 |

## 执行流程

### Step 0: 知识库检索（与 eat 联动）

> ⚠️ **调研前必做**：先检查 eat 是否已吸收相关知识，避免重复调研。
>
> 当前技能默认运行在 `Research Loop`：优先读取 `notes/research/`，并在存在重复失误信号时补读 `notes/lessons/`。

**强制动作**：
- [ ] 使用 Grep 搜索 `~/.claude/rules/` 中的相关主题
- [ ] 使用 Grep 搜索 `~/.claude/skills/` 中的相关技能
- [ ] 使用 Grep 搜索 `~/.claude/notes/research/` 中的相关主题
- [ ] 如问题涉及历史误判、反模式、设计返工，再搜索 `~/.claude/notes/lessons/`
- [ ] 如问题本身是结构/调用链设计，再搜索 `~/.claude/notes/design/`
- [ ] 记录已找到的相关知识

**检索方法**：
```bash
# 示例：搜索 "react-query" 相关知识
Grep pattern="react-query" path="~/.claude/rules"
Grep pattern="react-query" path="~/.claude/skills"
Grep pattern="react-query" path="~/.claude/notes/research"
Grep pattern="react-query" path="~/.claude/notes/lessons"
```

**证据记录**：
```
知识库检索记录：
- 检索关键词：...
- 命中结果：
  - rules-library/technique/xxx.md（相关度：高/中/低）
  - skills/xxx/SKILL.md（相关度：高/中/低）
  - notes/research/xxx.md（相关度：高/中/低）
  - notes/lessons/xxx.md（相关度：高/中/低）
  - notes/design/xxx.md（相关度：高/中/低）
- 是否需要补充调研：是/否
```

**处理策略**：

| 检索结果 | 处理方式 |
|---------|---------|
| **找到相关 research** | 复用已有证据与收敛结论，只补缺口 |
| **找到相关 lesson** | 优先沿用其决策框架，并检查 `Last Verified` 是否仍新鲜 |
| **找到相关 design** | 若本题属于结构/调用链设计，沿用其边界与约束 |
| **无相关内容** | 执行完整调研，考虑后续 eat 沉淀 |

### Step 1: 强制调研

> ⚠️ **不可跳过**：以下调研动作必须执行并记录证据。无证据 = 调研未完成。

#### 1.1 Repo 调研（必须执行）

**动作清单**：
- [ ] 使用 Glob/Grep 识别可能受影响的模块/文件
- [ ] 定位现有模式（类似功能、约定）
- [ ] 检查配置、依赖、构建工具

**证据记录**：
```
Repo 调研记录：
- 扫描路径：...
- 相关文件：file1.ts:10, file2.ts:45
- 现有模式：...
```

#### 1.2 业界调研（技术选型时必须执行）

**触发条件**（满足任一即执行）：
- 涉及新库/框架选型
- 涉及架构设计决策
- 涉及性能/安全关键路径
- 用户明确要求技术选型
- Step 0 知识库检索无相关内容

**强制动作**：
- [ ] 使用 WebSearch 搜索技术方案
- [ ] 记录搜索关键词
- [ ] 引用至少 2 个来源
- [ ] 对比 2-4 个方案
- [ ] 检查关键参考源的最近更新、changelog / release notes / migration notes
- [ ] 判断参考源是在强化原方案、弱化原方案，还是迁移到新主路径

**证据记录**：
```
业界调研记录：
- 搜索关键词："[技术名] best practices", "[技术名] comparison"
- 参考来源：
  1. 标题 - URL - 核心观点
  2. 标题 - URL - 核心观点
- 参考源演进检查：
  - 参考源：...
  - 最近版本/文档更新：...
  - 变化趋势：强化原方案 / 部分替代 / 明确废弃
  - 对当前决策的影响：...
- 方案对比：
  | 方案 | 优点 | 缺点 |
  |------|------|------|
  | A | ... | ... |
  | B | ... | ... |
```

**禁止行为**：
- ❌ 只写"我理解"或"根据经验"而不搜索
- ❌ 引用无来源的观点
- ❌ 跳过对比直接给出结论
- ❌ 只看教程/博客而不检查官方文档、changelog、release notes
- ❌ 只验证“现在能用”，不验证参考源是否正在迁移 away from 该方案

#### 1.3 参考源演进检查（架构/工作流决策时必须执行）

> ⚠️ **目的**：避免把“曾经正确但正在被替换”的方案误判为当前最佳实践。

**触发条件**（满足任一即执行）：
- 参考内容来自持续演进的项目/框架/工具链（如 Trellis、Claude Code、Next.js）
- 结论明显依赖某个外部项目的结构设计
- 用户要求“最佳实践”“当前最优”“结构优化方案”
- 该参考源过去 6-12 个月内仍持续发布

**强制动作**：
- [ ] 确认参考源的官方文档入口
- [ ] 查看最近 changelog / release notes / migration notes
- [ ] 区分“稳定核心”与“近期正在调整的边缘机制”
- [ ] 判断当前方案属于：`稳定沿用` / `收敛为新主路径` / `过渡期旧路径`

**决策框架**：

| 演进信号 | 含义 | 处理方式 |
|---------|------|---------|
| 官方文档持续强化同一结构 | 方案稳定 | 可作为主参考 |
| 新版本减少某类目录/命令自动注入 | 原方案边界收缩 | 仅保留为兼容层，不作为主路径 |
| changelog 明确迁移/重命名/按 project type 裁剪 | 方案正在收敛 | 采用新结构，旧结构标记过渡 |
| 官方示例已不再展示旧做法 | 旧做法可能退场 | 降低其权重，避免继续扩散 |

**输出要求**：
在调研结果中单列一节：

```markdown
### 参考源演进判断
- 参考源：...
- 当前主路径：...
- 旧路径是否仍推荐：是/否/仅兼容
- 对本次方案的影响：...
```

### Step 2: 复杂度评估

| 复杂度 | 标准 | 知识库检索 | 业界调研 | 行动 |
|--------|------|------------|----------|------|
| **Trivial** | 单行修复、错别字 | ❌ | ❌ | 直接实现 |
| **Simple** | 目标明确，1-2 文件 | ❌ | ❌ | 问 1 个确认问题 |
| **Moderate** | 多文件，有歧义 | ✅ | ⚡ 按需 | 轻量调研 + 2-3 问题 |
| **Complex** | 目标模糊，架构选择 | ✅ | ✅ | 完整调研 |

### Step 3: 扩展扫描（Diverge）

主动扩展思考：

| 维度 | 问题 |
|------|------|
| **未来演化** | 1-3 个月可能变成什么？保留什么扩展点？ |
| **相关场景** | 相邻功能应保持一致吗？有对等期望吗？ |
| **失败与边缘** | 冲突、重试、幂等性、回滚、验证？ |
| **上游演进** | 参考源是否已经出现新主路径？当前方案会不会很快过时？ |

### Step 4: 需求收敛（Converge）

**问题门控**：
1. **能推导吗？** → 自己调研 → **不问**
2. **是懒惰问题吗？** → 直接行动 → **不问**
3. **问题类型？** → 只问 **Blocking** 或 **Preference**

**Q&A 规则**：一次一个问题，优先多选格式

### Step 5: 知识沉淀（与 eat / lessons 联动）

> ⚠️ **调研后必评估**：判断是否需要将新发现写入 `notes/`、归并到 `notes/lessons/`，或进一步 eat 沉淀。

**沉淀判断标准**：

| 条件 | `notes/research` | `notes/lessons` | `/eat` |
|------|------------------|------------------|--------|
| 发现可复用的技术模式 | ⚡ 记录调研证据 | ✅ 如已上升为教训/决策 | ✅ 建议执行 |
| 发现架构设计最佳实践 | ⚡ 记录收敛过程 | ✅ 如适合长期提醒 | ✅ 建议执行 |
| 发现重复出现的错误或反模式 | ❌ | ✅ 优先归并到同主题 lesson | 按稳定度决定 |
| 仅项目特定配置 | ⚡ 可选 | ❌ | ❌ |
| 一次性解决方案 | ⚡ 可选 | ❌ | ❌ |

**lessons 归并流程**：

1. 先搜索 `notes/lessons/` 是否已有同主题 note
2. 若已有：更新原 note 的 `Last Verified`、`Source Cases`、新结论
3. 若没有：创建新的主题 lesson note，而不是按日期创建事件日志
4. 若同类案例持续增加：将状态提升为 `candidate-rule`
5. 若结论超过 90 天未验证：标记为 `stale`，引用前先复核

**沉淀触发**：
```
在调研结果输出后，追加：

## 知识沉淀建议
- 是否需要更新 notes/lessons：是/否
- lessons 目标主题：...
- 处理动作：更新已有主题 / 新建主题 / 标记 stale / 提炼 rule
- 是否需要 eat：是/否
- 理由：...
- 建议存储位置：rules-library/technique/ 或 rules-library/pattern/
```

## 反模式

| 反模式 | 正确做法 |
|--------|----------|
| 问用户可推导的内容 | 自己调研 |
| 呈现选项前让用户选择 | 先调研，再呈现具体选项 |
| 创建 PRD/任务文件 | 仅输出调研结果 |
| 只写"我理解"不搜索 | 使用 WebSearch 并引用来源 |
| 跳过调研直接提问 | 先完成调研证据记录 |
| 跳过知识库检索 | 先查 rules/skills，避免重复调研 |
| 调研完不沉淀 | 评估是否需要更新 lessons / 执行 eat |
| 发现同类教训仍新建日期笔记 | 先搜索旧 lesson，优先归并 |
| 只引用参考源的旧实现 | 检查其最近文档与 changelog，确认是否仍是主路径 |

## 调研未完成阻断

以下情况**禁止进入下一步**：

1. 未执行 Step 0 知识库检索
2. 技术选型任务但无业界调研记录
3. 业界调研无搜索关键词
4. 业界调研无引用来源（至少 2 个）
5. 涉及持续演进参考源但无“参考源演进判断”
6. 已命中相关 lesson 但未检查其新鲜度
7. 输出模板缺少"调研证据"字段

**阻断时的行动**：
- 告知用户："调研未完成，需要补充"
- 执行缺失的调研步骤
- 补全后再继续
