# Brainstorm - 调研与需求发现

**专注调研与需求发现，不涉及任务管理。**

```
用户描述任务 → 知识库检索 → 强制调研 → 需求收敛 → 输出调研结果 → 知识沉淀
```

## 核心原则

| 原则 | 含义 |
|------|------|
| **Research-first** | 技术选择必须先调研业界实践，再提出方案 |
| **Action-before-asking** | 能从代码/文档/研究中推导的，不问用户 |
| **Diverge→Converge** | 先扩展思考（未来演化/边缘情况），再收敛到 MVP |
| **One-question-at-a-time** | 一次只问一个关键问题 |
| **Knowledge-reuse** | 先查已有知识，再调研新知识，最后沉淀新发现 |

## 执行流程

### Step 0: 知识库检索（与 eat 联动）

> ⚠️ **调研前必做**：先检查 eat 是否已吸收相关知识，避免重复调研。

**强制动作**：
- [ ] 使用 Grep 搜索 `~/.claude/rules/` 中的相关主题
- [ ] 使用 Grep 搜索 `~/.claude/skills/` 中的相关技能
- [ ] 记录已找到的相关知识

**检索方法**：
```bash
# 示例：搜索 "react-query" 相关知识
Grep pattern="react-query" path="~/.claude/rules"
Grep pattern="react-query" path="~/.claude/skills"
```

**证据记录**：
```
知识库检索记录：
- 检索关键词：...
- 命中结果：
  - rules/technique/xxx.md（相关度：高/中/低）
  - skills/xxx/SKILL.md（相关度：高/中/低）
- 是否需要补充调研：是/否
```

**处理策略**：

| 检索结果 | 处理方式 |
|---------|---------|
| **找到完全匹配** | 直接引用现有知识，跳过该领域调研 |
| **找到部分相关** | 引用现有知识，仅补充缺失部分 |
| **无相关内容** | 执行完整调研，考虑后续 eat 沉淀 |

### Step 1: 强制调研

> ⚠️ **不可跳过**：以下调研动作必须执行并记录证据。无证据 = 调研未完成。

#### 1.1 Repo 调研（必须执行）

**动作清单**：
- [ ] 使用 Glob/Grep 识别可能受影响的模块/文件
- [ ] 定位现有模式（类似功能、约定）
- [ ] 检查配置、依赖、构建工具

**证据记录**：
```
Repo 调研记录：
- 扫描路径：...
- 相关文件：file1.ts:10, file2.ts:45
- 现有模式：...
```

#### 1.2 业界调研（技术选型时必须执行）

**触发条件**（满足任一即执行）：
- 涉及新库/框架选型
- 涉及架构设计决策
- 涉及性能/安全关键路径
- 用户明确要求技术选型
- Step 0 知识库检索无相关内容

**强制动作**：
- [ ] 使用 WebSearch 搜索技术方案
- [ ] 记录搜索关键词
- [ ] 引用至少 2 个来源
- [ ] 对比 2-4 个方案

**证据记录**：
```
业界调研记录：
- 搜索关键词："[技术名] best practices", "[技术名] comparison"
- 参考来源：
  1. 标题 - URL - 核心观点
  2. 标题 - URL - 核心观点
- 方案对比：
  | 方案 | 优点 | 缺点 |
  |------|------|------|
  | A | ... | ... |
  | B | ... | ... |
```

**禁止行为**：
- ❌ 只写"我理解"或"根据经验"而不搜索
- ❌ 引用无来源的观点
- ❌ 跳过对比直接给出结论

### Step 2: 复杂度评估

| 复杂度 | 标准 | 知识库检索 | 业界调研 | 行动 |
|--------|------|------------|----------|------|
| **Trivial** | 单行修复、错别字 | ❌ | ❌ | 直接实现 |
| **Simple** | 目标明确，1-2 文件 | ❌ | ❌ | 问 1 个确认问题 |
| **Moderate** | 多文件，有歧义 | ✅ | ⚡ 按需 | 轻量调研 + 2-3 问题 |
| **Complex** | 目标模糊，架构选择 | ✅ | ✅ | 完整调研 |

### Step 3: 扩展扫描（Diverge）

主动扩展思考：

| 维度 | 问题 |
|------|------|
| **未来演化** | 1-3 个月可能变成什么？保留什么扩展点？ |
| **相关场景** | 相邻功能应保持一致吗？有对等期望吗？ |
| **失败与边缘** | 冲突、重试、幂等性、回滚、验证？ |

### Step 4: 需求收敛（Converge）

**问题门控**：
1. **能推导吗？** → 自己调研 → **不问**
2. **是懒惰问题吗？** → 直接行动 → **不问**
3. **问题类型？** → 只问 **Blocking** 或 **Preference**

**Q&A 规则**：一次一个问题，优先多选格式

### Step 5: 知识沉淀（与 eat 联动）

> ⚠️ **调研后必评估**：判断是否需要将新发现的知识 eat 沉淀。

**沉淀判断标准**：

| 条件 | 是否 eat |
|------|----------|
| 发现可复用的技术模式 | ✅ 建议执行 `/eat` |
| 发现架构设计最佳实践 | ✅ 建议执行 `/eat` |
| 发现性能优化技巧 | ✅ 建议执行 `/eat` |
| 仅项目特定配置 | ❌ 不需要 |
| 一次性解决方案 | ❌ 不需要 |

**沉淀触发**：
```
在调研结果输出后，追加：

## 知识沉淀建议
- 是否需要 eat：是/否
- 理由：...
- 建议存储位置：rules/technique/ 或 rules/pattern/ 或 rules/best-practice/
```

**调研结果交付后，转交给实现工作流（如 Plan 阶段）处理。**

## 反模式

| 反模式 | 正确做法 |
|--------|----------|
| 问用户可推导的内容 | 自己调研 |
| 呈现选项前让用户选择 | 先调研，再呈现具体选项 |
| 创建 PRD/任务文件 | 仅输出调研结果 |
| 只写"我理解"不搜索 | 使用 WebSearch 并引用来源 |
| 跳过调研直接提问 | 先完成调研证据记录 |
| 跳过知识库检索 | 先查 rules/skills，避免重复调研 |
| 调研完不沉淀 | 评估是否需要 eat 沉淀 |

## 调研未完成阻断

以下情况**禁止进入下一步**：

1. 未执行 Step 0 知识库检索
2. 技术选型任务但无业界调研记录
3. 业界调研无搜索关键词
4. 业界调研无引用来源（至少 2 个）
5. 输出模板缺少"调研证据"字段

**阻断时的行动**：
- 告知用户："调研未完成，需要补充"
- 执行缺失的调研步骤
- 补全后再继续
